## ЧАСТЬ 3. Построение RAG-пайплайна 
### Задание 3. Допустим, у вас есть 10 PDF-файлов с внутренними регламентами компании.
Опишите архитектуру RAG-пайплайна:
1. Как вы разобьёте документы на чанки (размер, перекрытие)?

    > Для структурированных регламентов - паттерн Parent-Child:
    > Child (2-3 абзаца, ~300 токенов) - для точного векторного поиска. Маленький размер = выше precision retrieval.

    > Перекрытие: 50 токенов (~15%)
    > Стратегия: RecursiveCharacterTextSplitter с кастомными separators (\n\n → \n → . )

    > Parent (целый раздел/статья, ~1500 токенов) - передаётся в LLM.
    > Перекрытие: 100 токенов
    > Каждый Child ссылается на Parent через parent_id в метаданных.

    > Для более сложного парсинга, например табличных данных, возможно использовать специальные инструменты, например LlamaParser или Docling

2. Какую модель для эмбеддингов вы используете (локальную или через API)?

      >  Для эмбеддингов я использую bge-m3 локально. Bge-m3 хорошо себя показывает на русском языке, открытая лицензия, размерность 1024 против 768 у nomice, весит не много, на gpu занимает около 1-2 гб видеопамяти. 

      > Кроме того, модель поддерживает dense, sparse и multi-vector представления, что позволяет использовать более продвинутые сценарии поиска при необходимости.

      > ЛОКАЛЬНО так как использование локальной embedding-модели позволяет:
          > - избежать отправки внутренних документов во внешний API;
          > - снизить стоимость эксплуатации;
          > - уменьшить задержку поиска;
          > - обеспечить работу даже без доступа к Интернету. 

3. Какую векторную БД выберете и почему (FAISS, Chroma, Qdrant)?

    > Qdrant сейчас — стандарт в индустрии. Он высокопроизводительный, поддерживает гибридный поиск, фильтрацию по метаданным, удобен в использовании, имеет хорошую документацию.

4. Как будет происходить поиск релевантных чанков по запросу пользователя?

    > Пользователь пишет запрос, если система решает искать ответ в БД, то запрос с помощью тойже bge-m3 переводится в вектор и по этому вектору ищутся наиболее похожие чанки из Qdrant. Возвращается n-количество (указываем сами) подходящих ответов.

5. Как вы передадите найденные чанки в промпт агента?

    > Процесс можно описать так:
    > - Агент получает вопрос пользователя и анализирует его.
    > - Если вопрос касается документации, агент принимает решение вызвать инструмент search_knowledge_base.
    > - Инструмент обращается к векторной БД (Qdrant), получает релевантные чанки и возвращает их агенту в виде отдельного сообщения (Function/Tool Message) в историю диалога.
    > - Возвращаемые чанки содержат не только текст, но и метаданные (название документа, раздел, страница).
    > - На следующей итерации агент видит контекст из Tool Message, анализирует его и генерирует финальный ответ пользователю, опираясь на этот контекст и строго цитируя метаданные источника.

Примечание
> Ещё в качестве улучшения я бы использовал rerank модель (например, надёжный вариант bge-rerank). Rerank может дать +5-10% качества на стадии поиска релевантных чанков. VRAM  с выбранными моделями позволяет это сделать.

### Задание 5. На вашем сервере 24 ГБ VRAM, но в пиковые часы одновременно работают 3 агента с разными задачами.
Что вы сделаете:

• Для оптимизации использования памяти (kv-cache, batch-size).

   > Я бы использовал vllm. Этот движок специально предназначен для более менее легкого запуска моделей с упором на производительность. В нём есть две ключевые особенности: Page attention и continuous batching. Благодаря этим улучшениям пропускная способность сильно возрастает. Flash attention  здесь используется по умолчанию

   > TensorRT даёт ещё большую оптимизацию, но сложно настраивается. 

   > В ollama можно подключить flash-attention для ускорения
   > OLLAMA_FLASH_ATTENTION=1
